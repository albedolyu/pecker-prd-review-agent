# 决策清单和试用节奏

## 会中需要确认的决策

### 1. 试用范围

建议决策：先开放给 2-3 位 PM 代表，试用 1 周后再扩到 5-6 人。

不建议一上来全员开放。当前更适合内部 Beta，而不是正式生产系统。

### 2. 部署方式

建议决策：部署到内部服务器或固定办公机器，对同事提供 Web 入口。

不建议让同事轮流使用我的电脑。个人电脑适合本地开发和临时演示，不适合作为团队工具入口。

### 3. 数据边界

建议决策：试用期允许指定 PM 在内网环境使用未脱敏真实 PRD。

明确边界：

- 未脱敏真实 PRD 只能在内网工具中使用。
- 评审报告、修订建议包和修订稿草案只能内部流转。
- 不通过飞书群、邮件群或公网网盘外发未脱敏 PRD 导出文件。
- 合同原件、身份证号、手机号、银行账号等强敏感个人信息仍需最小化处理。

### 4. 成本上限

建议决策：上线前启用三档硬上限。

- 单次评审上限：建议先按 3 美元级别控制。
- 单人单日上限：建议先按 30 美元级别控制。
- 项目月度上限：建议先按 500 美元级别控制，真实使用后再调整。

超过额度时，UI 要明确提示，不做静默失败。

### 5. 反馈机制

建议决策：PM 代表每条建议尽量做确认、误报、遗漏标记。

每周复盘一次：

- 哪些意见真的减少了返工。
- 哪些意见是误报。
- 哪些重要问题仍然漏掉。
- 哪些资料库或规则需要维护。

## 建议试用节奏

### Day 0-1：服务端准备

- 配置团队版 OpenAI 兼容 API 环境变量。
- 确认 API key 不进前端、不进日志、不进文档。
- 启动后端、前端和健康检查。
- 跑一次 demo review 和一次未脱敏真实 PRD smoke。
- 确认失败页、降级页、额度提示都能被用户看懂。

### Week 1：2-3 位 PM 代表

目标：验证“愿不愿意用”和“是否减少返工”。

建议样本：

- 10-15 份真实 PRD。
- 同时覆盖新功能、流程改动、字段口径、风险边界类需求。
- 每位 PM 至少完成 3 次评审和 10 条反馈标记。

### Week 2：扩到 5-6 人

目标：验证并发、稳定性和成本。

关注指标：

- 5-6 人同时发起评审时，不出现认证互踢或大量 401。
- 运行失败时，UI 能明确说明失败原因和下一步。
- 单次成本和日成本在预算闸内。
- PM 是否继续愿意把它放进真实工作流。

### Week 3：决定是否扩大

建议判断标准：

- Productive session 逐步接近 85% 以上，后续目标是稳定超过 90%。
- 反馈标记率超过 60%。
- 严重数据合规事故为 0。
- PM 能明确说出至少 2 类被减少的返工问题。

## 会后产出

会后需要形成一页结论：

- 是否批准内部 Beta。
- 试用 PM 名单。
- 部署机器或服务器。
- 数据边界口径。
- 预算上限。
- 第一周复盘时间。
