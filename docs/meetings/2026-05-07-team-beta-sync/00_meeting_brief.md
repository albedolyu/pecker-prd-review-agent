# 啄木鸟内部团队版会前对齐稿

## 一句话定位

啄木鸟是 PRD 提交前的 AI 质量门：帮助 PM 在交给研发、测试或评审会之前，提前发现业务完整性、字段口径、使用体验和实现风险问题。它不替代 PM 决策，也不替代研发、测试、法务的最终判断。

## 今天希望达成的结论

建议进入“受控内部 Beta”。

可试用的原因：

- Web 端和 CLI 都已经可用，当前主线已收口到 `main`。
- 模型路由已切到 OpenAI 兼容 API 路径，团队版不再依赖个人 OAT/CLI session。
- 本地验证过 5-6 人级别并发：配置为 `PECKER_MAX_CONCURRENT=6`，模型调用闸为 `PECKER_MAX_CONCURRENT_MODEL_CALLS=3`。
- 后端测试基线已跑到 `1344 passed`，Web 端也完成过测试、类型检查和构建验证。
- 压测 session 已打标并从产品指标里排除，避免压力测试污染真实体验指标。

不建议直接全量开放的原因：

- 最新真实产品指标还不是 90% 稳态：近期有效一致率约 72%，还需要真实 PM 使用数据继续校准。
- 仍需要服务端部署、备份、预算上限、数据合规边界和 on-call 预案。
- PRD 可能包含未公开功能、价格、客户名或战略信息，试用期必须要求先脱敏。

## 当前产品能力

- 新建 PRD 评审：上传或粘贴 PRD，选择资料库和轻/深评审模式。
- 盲区预检：先发现材料缺口和高风险输入，不急着给结论。
- 多评审员并行：业务、数据、体验、风险四个方向分别产出意见。
- 交叉校验：苍鹰复核多位评审员之间的冲突、误报和遗漏。
- 运行质量检查：在交给 PM 前确认本次结果是否可靠。
- PM 逐条确认：PM 可以确认、驳回或补充意见，沉淀为后续规则权重。
- 评审记录和质量看板：用于回看历史、观察完成率和失败类型。

## 需要领导拍板

1. 是否同意进入 2-3 位 PM 代表的小范围 Beta。
2. 是否允许在内部服务器部署，而不是继续依赖个人电脑。
3. 试用期的数据边界：只允许脱敏 PRD，不上传真实客户名、真实价格、未公开战略材料。
4. 预算上限：建议先设单次、单人单日、项目月度三档硬上限。
5. 反馈机制：PM 代表每条评审建议都尽量做“确认/误报/遗漏”标记，每周复盘一次。

## 建议会议结论

先批准受控 Beta，不承诺 SLA，不做全员推广。用 1-2 周真实 PRD 试用数据判断是否扩大范围；验收重点不是“AI 看起来聪明”，而是 PM 是否愿意持续使用、是否减少返工、误报是否可控。
