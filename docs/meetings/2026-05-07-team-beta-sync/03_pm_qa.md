# PM 代表 Q&A

## 现在能不能给同事用？

可以，但建议按“内部 Beta”给同事使用，而不是按正式系统全员开放。

推荐先找 2-3 位 PM 代表，用脱敏 PRD 试 1 周。确认稳定性、反馈闭环和成本后，再扩到 5-6 人。

## 同事需要用我的电脑吗？

不建议。

本地电脑适合开发和演示。团队使用应该部署到内部服务器或固定办公机器，然后同事通过 Web 地址访问。

## Web 端和 CLI 都可以用吗？

可以。

Web 端适合 PM 日常使用；CLI 更适合我做本地排障、批量验证和压测。给同事试用时，建议只开放 Web 端。

## 这个工具会替代正式 PRD 评审吗？

不会。

啄木鸟是正式评审前的质量门，帮助 PM 提前发现材料缺口和返工风险。最终判断仍然由 PM、研发、测试和相关负责人完成。

## 为什么有时输出的问题数量会变少？

这是预期优化方向。

系统会过滤重复项、低置信项、互相冲突的建议，把更值得 PM 处理的问题留下来。目标不是让 AI 输出更多，而是让 PM 更快看到真正该改的地方。

## 当前模型是不是都强制用 GPT-5.5？

不建议全部强制 GPT-5.5。

团队版走 OpenAI 兼容 API 路由，关键复核和深评审可以用更强模型，部分轻量任务可以走更快、更便宜的模型。这样体验更顺，成本也更可控。需要时可以临时调高模型，但不建议把所有路径都固定到最高成本模型。

## 如果某个评审员失败怎么办？

系统会按降级方式处理：

- 部分评审员失败：展示已完成维度，标明缺失维度，允许重试。
- 复核层失败：展示原始评审意见，并提示“未交叉校验”。
- 全部超时：停止无限加载，保存原始 PRD 和错误信息，提示重新评审或查看错误。

## 为什么指标不是 100%？

这是 AI 工具进入真实工作流前必须面对的部分。

当前我们更关注三件事：失败是否可见、是否能恢复、真实 PM 是否愿意继续用。Beta 的目标是把 productive session 从当前水平持续提升到 85%-90% 以上，而不是在演示里只展示一次完美结果。

## 压测数据会不会污染真实指标？

不会。

现在压测 session 已经打标，状态报告默认排除压测数据。真实产品指标会更接近 PM 实际使用体验。

## PRD 数据安全吗？

试用期要按保守口径处理。

不要上传真实客户姓名、真实价格、未公开战略和其他不能外发的材料。API key 只放服务端，前端不会接触 key，也不能把 `.env` 发给同事。

## 成本会不会失控？

上线前要启用预算闸。

建议先设置单次评审、单人单日、项目月度三档硬上限。超过额度时 UI 明确提示，不静默调用模型。

## PM 代表需要做什么？

试用期最重要的是标反馈。

每条建议尽量标记为：

- 确认：这个问题确实有价值。
- 误报：这个问题不成立。
- 遗漏：AI 没发现但 PM 发现了。

这些反馈会影响后续规则权重，是啄木鸟能不能越用越准的关键。
