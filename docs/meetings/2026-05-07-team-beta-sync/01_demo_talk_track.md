# 现场演示讲稿

## 建议时长

总时长 20-30 分钟：

- 3 分钟：说明背景和定位。
- 8-10 分钟：演示 UI 全流程。
- 7-10 分钟：讨论试用边界和风险。
- 5 分钟：确认会后动作。

## 开场话术

啄木鸟的目标不是替 PM 写 PRD，也不是替评审会下结论。它做的是 PRD 交付前的质量检查，把常见返工点提前暴露出来，让 PM 带着更完整、更一致的材料进入正式协作。

今天我希望大家重点看三件事：

1. 流程是不是 PM 能理解和愿意用。
2. 输出是不是能帮助减少返工，而不是制造额外工作。
3. 内部 Beta 的风险边界是否可接受。

## 演示路径

### 1. 登录和入口

讲法：

这是内部工具，不开放公网注册。登录后进入“新建一次 PRD 评审”，可以上传 PRD，也可以用演示模式看完整流程。

要强调：

- 后续同事不需要用我的电脑，只要部署到内部服务器并分配账号即可。
- 试用期建议只开放给 PM 代表，避免还没稳定就被全员按正式系统预期使用。

### 2. 上传 PRD 或使用演示模式

讲法：

PM 上传 PRD 后，先选资料库和评审模式。轻评审适合日常预检，深评审适合正式提交前。

要强调：

- 试用期允许指定 PM 在内网上传未脱敏真实 PRD。
- 演示会议仍建议使用内置 demo，避免会议现场投屏暴露真实 PRD。

### 3. 盲区预检

讲法：

系统第一步不是直接挑刺，而是先判断材料是否足够评审。例如目标是否清楚、关键字段是否缺口、流程和边界条件是否足够完整。

要强调：

- 如果材料本身不够，系统会提醒先补材料。
- 这一步的价值是减少“拿着不完整 PRD 进入正式评审”的情况。

### 4. 多评审员并行

讲法：

这里可以看到四个方向同时评审：业务完整性、字段口径、使用体验、实现风险。它们彼此不直接互相影响，最后统一交给复核层。

要强调：

- 不是一个大模型一口气输出，而是按 PM 真实评审维度拆开。
- 每个方向有边界，避免输出发散。
- 团队版已按 5-6 人并发做过本地压测配置。

### 5. 苍鹰交叉校验

讲法：

苍鹰不新增大量意见，它主要检查误报、遗漏和多位评审员之间的冲突。这样最终给 PM 的结果会更少、更集中。

要强调：

- 输出变少不是能力下降，而是把重复项、低置信项和互相冲突的内容过滤掉。
- PM 看到的应该是更可执行的问题，而不是一堆 AI 展示欲很强的长列表。

### 6. 运行质量检查

讲法：

在进入 PM 确认前，系统会判断本次评审是否可靠。如果有评审员失败、超时或复核缺失，会标明降级状态。

要强调：

- 失败不会直接白屏或让 PM 空等。
- 部分失败时会给出已完成维度，并允许重试。

### 7. PM 逐条确认和反馈回流

讲法：

PM 最后逐条处理评审建议，可以确认、驳回或补充。这些反馈不是只存在本次报告里，而是会影响后续规则权重和评审质量。

要强调：

- 反馈是啄木鸟变好的关键资产。
- 如果 PM 不标反馈，系统只能靠离线经验，很难贴合团队真实口径。

### 8. 报告和历史记录

讲法：

报告用于会后归档和协作。历史记录和质量看板用来观察工具是否真的在变好，包括完成率、失败率、误报情况。

要强调：

- 质量看板不是为了证明工具完美，而是为了暴露问题并持续校准。
- 压测数据已经打标，不会混进真实产品指标里。

## 现场异常兜底

如果网络或模型网关现场不稳定：

- 直接切演示模式看 UI 全流程。
- 说明真实评审依赖 LLM 网关，正式试用前会部署在内部服务器并做健康检查。
- 不在会议现场投屏敏感 PRD 内容，真实 PRD 试用放到会后内网环境跑。
