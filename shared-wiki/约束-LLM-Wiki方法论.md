# LLM Wiki 方法论

> 来源：llm-wiki.md — 一种用 LLM 构建和维护个人知识库的模式。
> 本页为精简中文摘要，供鸮鹦迭代设计参考。

## 核心理念

传统 RAG 每次查询都从原始文档重新检索拼凑，知识不积累。LLM Wiki 的做法不同：LLM **增量构建并持续维护一个持久化 wiki**——一组结构化、互相链接的 markdown 文件。每添加一个新来源，LLM 就将其知识编译进 wiki，更新实体页、修正矛盾、强化综合判断。知识被编译一次、持续更新，而非每次查询时重新推导。

**关键区别：wiki 是一个持久的、复利增长的知识制品。**

---

## 三层架构

| 层级 | 职责 | 所有权 | 鸮鹦映射 |
|------|------|--------|----------|
| **Raw Sources** | 不可变的原始资料（文章、PRD、数据文件） | 人类策展 | 已实现：PRD 文件由用户提供，鸮鹦只读不改 |
| **Wiki** | LLM 生成的 markdown 页面集合（摘要、实体页、概念页、对比分析） | LLM 拥有 | 已实现：shared-wiki 目录，鸮鹦负责创建和维护全部页面 |
| **Schema** | 告诉 LLM wiki 的结构规范、命名约定、工作流程的配置文件 | 人 + LLM 共同演化 | 部分实现：命名前缀（概念-/场景-/约束-/决策-/实体-）和 frontmatter 规范已有，但缺少完整的 Schema 文档 |

---

## 三种操作

### 1. Ingest（摄入）

用户将新资料加入原始集合，LLM 处理流程：
- 阅读资料，与用户讨论要点
- 在 wiki 中写入摘要页
- 更新 index
- 更新相关的实体页和概念页（一个来源可能触达 10-15 个页面）
- 在 log 中追加记录

**鸮鹦状态：部分实现。** 当前 review 流程会产出 wiki 页面并更新 index/log，但缺少"跨页面级联更新"能力——新资料的知识只写入新页面，不会自动回溯更新已有页面。

### 2. Query（查询）

用户提问，LLM 流程：
- 先读 index 定位相关页面
- 读取具体页面内容
- 综合生成带引用的回答
- **好的回答可以反向归档为新 wiki 页面**，让探索过程也产生复利

**鸮鹦状态：未实现。** 当前鸮鹦是批处理模式（review → 产出页面），不支持交互式查询。

### 3. Lint（健康检查）

定期让 LLM 检查 wiki 健康度：
- 页面间的矛盾
- 被新资料取代的过时声明
- 没有入站链接的孤岛页面
- 被提及但缺少独立页面的重要概念
- 缺失的交叉引用

**鸮鹦状态：已实现。** kakapo_dream.py 的 Phase 3（Consolidate）和 Phase 4（Prune/Index）覆盖了大部分 Lint 功能：孤岛检测、过时判定、frontmatter 缺失检查、重复页面相似度检测。

## Index + Log 规范

### index.md（内容导向）
- wiki 中所有页面的目录：链接 + 一行摘要 + 可选元数据
- 按类别组织（实体、概念、来源等）
- 每次 ingest 后更新
- 查询时 LLM 先读 index 定位页面，再深入阅读——在中等规模（~100 来源、几百页面）下效果良好，无需向量检索

**鸮鹦状态：已实现。** index.md 按前缀分类，包含链接和摘要。

### log.md（时间线导向）
- 仅追加的操作记录，记录时间和操作类型
- 推荐格式：`## [YYYY-MM-DD] 操作类型 | 资料名称`，便于 grep 解析
- 提供 wiki 演化的时间线

**鸮鹦状态：已实现。** log.md 格式与方法论推荐一致。

## 鸮鹦能力差距总结

| 能力 | 方法论要求 | 鸮鹦现状 | 优先级建议 |
|------|-----------|----------|-----------|
| 三层架构 | Raw / Wiki / Schema | Raw + Wiki 已有，Schema 缺正式文档 | P2 |
| Ingest 级联更新 | 新资料触达 10-15 个已有页面 | 只写新页面，不回溯更新 | **P1** |
| Query 交互 | 基于 index 的交互式问答 | 无，仅批处理 | P1 |
| Query 归档 | 好回答反向存入 wiki | 无 | P2 |
| Lint 健康检查 | 矛盾/过时/孤岛/缺失检测 | 大部分已覆盖 | 已完成 |
| CLI 搜索工具 | wiki 规模大时需要 BM25/向量检索 | 无，依赖 index 线性扫描 | P3（当前规模够用） |

## 设计启发

1. **知识复利**：wiki 的价值在于编译而非检索。鸮鹦下一步应让每次 review 的产出能回溯丰富已有页面。
2. **Schema 显式化**：将当前隐含在代码中的命名规范、前缀约定、frontmatter 要求提取为独立的 Schema 文档，便于新 agent 理解和遵守。
3. **Query 能力**：支持用户基于 wiki 提问，是从"工具"到"知识伙伴"的关键跃迁。
