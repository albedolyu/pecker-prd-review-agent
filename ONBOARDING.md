# 啄木鸟 PRD 评审 — 新人 5 分钟上手

> 啄木鸟专门在树上找虫。PRD 就是那棵树，bug 就是虫。
> 这份文档面向团队 PM，第一次用啄木鸟自查 PRD 看这个就够。

> Beta 阶段 · PM 内部工具。评审结果用于预审参考，不替代 PM / 研发 / 法务的最终判断；反馈直接发到飞书群「啄木鸟反馈」。
> 不要上传含真实客户姓名、真实价格、未公开战略或未经脱敏客户材料的 PRD；必要时先脱敏再评审。

---

## 谁要用？

- **PM 写完 PRD 想自查** → 跑啄木鸟 → 拿到改进 checklist，按改完再发评审
- **PM 想避免常见坑** → 看历史 finding 累积下来的 Learnings 知识沉淀
- **不是 PM** → 你大概率不需要看这份；开发同学请直接看 [DEV.md](./DEV.md)

---

## 怎么访问？

### Web 版（推荐，零安装）

1. 打开浏览器：`http://<服务器内网 IP>:3000`
2. 第一次进会跳到 `/login`
3. 输入：
   - **评审人姓名**：填中文名，会进报告署名 + 审计日志
   - **团队密码**：问工具负责人或飞书群「啄木鸟反馈」
4. 登录后 JWT cookie 8 小时有效，关掉浏览器再开不用重输

### CLI（开发者 / 自动化）

如果你在服务器上跑 `python run_session.py`，看 [DEV.md](./DEV.md) 和 [啄木鸟_使用指南.md](./啄木鸟_使用指南.md)。

## 5 分钟评审一份 PRD（Web）

| 阶段 | 你做的 | 系统做的 | 时长 |
|---|---|---|---|
| Phase 0 上传 | 拖入 PRD 文件 / 粘贴内容；选 workspace（业务线）；选 profile | 文件落盘到 `workspace-<业务线>/prd/` | <1 分钟 |
| Phase 1 预检 | 等 | 知识库覆盖扫描，分 wiki 强相关 / 弱相关 / 知识盲区三栏 | ~15 秒 |
| Phase 2 评审 | 等 / 看进度 | 4 位编辑（责编 / 审校 / 技术编辑 / 数据核对员）并行评审，可选苍鹰交叉校验 | 5-15 分钟 |
| Phase 3 确认 | 按职能 Tab 浏览改进项，每条点 接受 / 拒绝 / 改写 | 实时算啄伤度（100 = 重写 / 50 = 该改 / 0 = OK） | 自定 |
| Phase 4 报告 | 下载 .md / 保存到 wiki / 推飞书群 | 生成报告：`PRD_开发任务_<日期>_<你名字>.md` | <30 秒 |

### 选 profile（重要）

| profile | 展示什么 | 适合 |
|---|---|---|
| **chill**（默认） | 只展示 `must`（必须改）+ 高置信 `should`（建议改），隐藏 `could`（可改） | 发版前自查、avoid noise |
| **strict** | 全展示含 nitpick | 架构 review、新规则 calibration |

> 默认就是 chill，觉得评审太严就保持默认；觉得太宽就切 strict。
> CLI 用法：`python run_session.py xxx --profile chill` 或 env `PECKER_PROFILE=chill`。

## 报告怎么读？

### 顶部看「啄伤度」

| 啄伤度 | 含义 | 行动 |
|---|---|---|
| 100 | 几乎重写 | 按 finding 全改完再发评审 |
| 50-90 | 该改 | 至少处理 must 和高置信 should |
| 0-50 | OK | 复查 must 即可发版 |

### 按 PRD 章节分组

每条 finding 三块结构：

- **【位置】** ：PRD 哪一章 / 哪一段
- **【问题】** ：违反了什么、影响什么
- **【怎么改】** ：具体修改建议（不是抽象原则）

### 末尾 metadata（灰色小字）

`<sub>` 标签，含位置 / 严重度 / 依据类型 / 规则号（RC-001 / R-005 等）。后续要给反馈用规则号定位。

## 常见问题

| 问题 | 解决 |
|---|---|
| 评审太严了，太多 nitpick | 切 `--profile chill`（默认就是） |
| 评审太宽了，没啄到我知道的坑 | 切 `--profile strict`，或者考虑加规则（找规则维护人） |
| 我觉得这条是误报 | 飞书群「啄木鸟反馈」@啄木鸟机器人留言：`R-005 是误报，理由是 X`。系统会记录到 Learnings，下次自动避开（信鸽 v2，累计 reject ≥ 3 自动升级 learning） |
| 评审跑挂了 / 报错 | 找工具负责人；附评审 ID（看浏览器 URL）和报错截图 |
| 想加新规则 | 找规则维护人或工具负责人提需求；规则在 `workspace-*/review-rules/review-checklist.yaml` |
| 报告找不到了 | 在 `workspace-<业务线>/output/` 下，按日期排序 |
| 知识盲区栏显示我业务的资料没覆盖 | 把 PDF / Excel / 资料丢到 `workspace-<业务线>/raw/`，下次评审会自动入 wiki |

- [ ] 我能登录 Web 版（浏览器进 `http://<内网>:3000`）

- [ ] 我知道挂了找谁

- [ ] 我把团队 ≥ 1 份真实 PRD 跑过完整流程

- [ ] 我看过 `workspace-<业务线>/wiki/` 的知识库 index 至少一次
- [ ] 我给至少一条 finding 反馈过（接受 / 拒绝 / 改写）
