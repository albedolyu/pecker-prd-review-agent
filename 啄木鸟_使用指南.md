# 啄木鸟 (Pecker) — 使用指南

> 啄木鸟专门在树上找虫。PRD 就是那棵树，bug 就是虫。
> 十只鸟各司其职：评审团并行找虫，苍鹰交叉校验，杜鹃验证质量，鸮鹦整理知识库，信鸽从下游带回反馈，伯劳审核产出。

## 前置条件

- Python 3.10+
- Anthropic API Key（支持中转站）

---

## 快速开始（3 步）

**第一步：克隆共享知识库**

```bash
git clone https://github.com/xinshu001/prd-wiki.git
```

记下克隆到的本地路径，下一步填入 `.env`。

**第二步：配置环境变量**

```bash
cp .env.example .env
```

打开 `.env` 填入以下内容：

```bash
# 走本地 Claude Code CLI，零 API key（需先 `claude login`）
USE_CLAUDE_CODE=1
WIKI_PATH=填入你的本地路径
REVIEWER=你的名字
FEISHU_WEBHOOK=            # 可选
```

**第三步：安装依赖并启动**

```bash
pip install -r requirements.txt
python run_session.py PRD名称
```

评审前确保：业务资料放入 `raw/`，待评审 PRD 放入 `prd/`。

## Web 版使用

适合不使用 CLI 的同事，打开浏览器即可评审 PRD。**目前有两个版本并行**：

- **Next.js 新版(推荐)** `http://localhost:3000` — Phase B-D 重构后的版本，编辑部术语，支持多人并发
- **Streamlit 旧版(迁移期 fallback)** `http://localhost:8501` — 单用户，原版 Agent 界面

### 启动(新版)

管理员跑起后端 + 前端：

```bash
# 终端 1：FastAPI 后端
uvicorn api.main:app --reload --port 8000

# 终端 2：Next.js 前端
cd web && pnpm dev
```

> 如果你是最终用户(不是管理员)，直接访问团队部署的地址即可。

### 首次登录

1. 打开 `http://localhost:3000`，自动重定向到 `/login`
2. 输入**评审人姓名**(自己的名字，会记录在报告署名 + 审计日志)
3. 输入**团队密码**(由管理员通过 `PECKER_WEB_PASSWORD` env var 配置)
4. 登录成功后自动进入 `/review`，JWT cookie 有效期 8 小时

### 五阶段评审流程

| Phase | 阶段 | 你要做的事 |
|---|---|---|
| 0 | 上传 | 拖入 PRD 文件或粘贴内容，选 workspace，选模式(快速/严格)，可选补充说明 |
| 1 | 预检 | 等待 ~15s，看 wiki 强相关 / 弱相关 / 知识盲区三栏结果 |
| 2 | 评审 | 观察 4 位编辑(责编 / 审校 / 技术编辑 / 数据核对员)并行跑，~90-150s，严格模式后接终审 |
| 3 | 确认 | 按职能 Tab 浏览所有改进项，每条点 ✅接受 / ❌拒绝 / ✏️改写，系统会算"啄伤度" |
| 4 | 报告 | 下载 .md 本地副本 / 保存到 workspace/wiki / 推送飞书群(只读用户后两者禁用) |

### 关键概念

- **Workspace**: 每个业务方向一个目录(`workspace-对外投资`、`workspace-侵权软件` 等)，每个 workspace 有自己的 `prd/` `wiki/` `raw/` `output/`
- **评审人姓名**: 可以用中文，会进 JWT payload，所有审计事件、报告署名都会带
- **只读用户**: 管理员通过 `PECKER_READONLY_USERS=张三,李四` 配置，这些用户能完整走完评审流程但不能保存到 wiki / 推送飞书
- **草稿恢复**: 评审到一半浏览器崩溃，再次打开会提示"发现未完成的评审草稿"，一键恢复或丢弃
- **编辑部术语**: UI 里的主编 / 责编 / 审校 / 技术编辑 / 数据核对员 / 终审就是内部代码里的啄木鸟 / 织布鸟 / 猫头鹰 / 渡鸦 / 鸬鹚 / 苍鹰。`/about` 页有完整的 10 鸟品牌故事

### 并发上限

公共 Claude CC 账号有 rate limit，所以后端用 `asyncio.Semaphore` 限制同时进行的评审数量，默认 2(可通过 `PECKER_MAX_CONCURRENT` 调整)。超额的请求会排队。

### Streamlit 旧版启动

仍然可用作 fallback，单用户场景：

```bash
streamlit run app.py --server.port 8501
```

会自动打开 `http://localhost:8501`。操作流程和新版相同，但 UI 是旧的、只支持一个用户。

## CLI 参数

```bash
# 自动路由（推荐）— Haiku 判断 PRD 复杂度，自动选模型
python run_session.py PRD名称

# 指定评审人
python run_session.py PRD名称 --reviewer zhangsan

# 指定工作目录
python run_session.py PRD名称 --workspace /path/to/workspace

# 强制指定模型
python run_session.py PRD名称 --model opus     # 深度评审
python run_session.py PRD名称 --model sonnet   # 标准评审
python run_session.py PRD名称 --model haiku    # 格式检查

# 关闭并行评审（单 agent 串行模式）
python run_session.py PRD名称 --no-parallel
```

## 独立工具

### 信鸽 — 反馈闭环

AI Coding 完成后运行，从代码中采集反馈信号并更新规则权重：

```bash
python feedback.py \
  --code-dir /path/to/code \
  --report output/PRD_改动报告_xxx.md \
  --prd prd/xxx.md \
  --scope keyword1,keyword2
```

`--scope` 可选，限定扫描范围（文件路径关键词过滤）。

### 杜鹃 — 评审质量验证

```bash
# 完整评测（需要测试用例文件）
python cuckoo_eval.py \
  --report output/PRD_改动报告_xxx.md \
  --test-case eval/test_cases/xxx.json \
  --workspace ./workspace

# 从已有报告生成测试用例
python cuckoo_eval.py \
  --report output/PRD_改动报告_xxx.md \
  --generate-test-case \
  -o eval/test_cases/xxx.json
```

### 鸮鹦 — Wiki 知识库维护

```bash
# 完整整理（扫描 → 修复 → Diff → 重建索引）
python kakapo_dream.py --wiki-path ./workspace/wiki

# 只扫描，不修复
python kakapo_dream.py --wiki-path ./workspace/wiki --scan-only
```

### 苍鹰 — 交叉校验

```bash
python goshawk_advisor.py --prd prd/xxx.md --report output/PRD_改动报告_xxx.md
```

### 伯劳 — 质量门禁

```bash
python shrike_review.py --workspace ./workspace
python shrike_review.py --workspace ./workspace --output review_report.md
```

检查内容：安全红线、wiki 一致性（断链/命名/frontmatter）、报告完整性。

## 环境变量说明

| 变量 | 必填 | 说明 |
|------|------|------|
| `USE_CLAUDE_CODE` | 是 | 设为 `1` 走本地 Claude Code CLI（需先 `claude login`） |
| `WIKI_PATH` | 是 | 共享知识库本地路径（git clone 的路径） |
| `REVIEWER` | 是 | 评审人名字，用于分支命名和报告署名 |

## 安全机制

| 保护 | 说明 |
|------|------|
| **文件权限** | `raw/`、`prd/`、`review-rules/` 只读 |
| **bash 白名单** | 只允许 git 和 pwd 命令 |
| **危险命令拦截** | `rm -rf`、`git push -f`、`curl` 等直接拒绝 |
| **push 确认** | `git push` 需要手动确认 y/n |
| **路径围栏** | 所有文件操作限制在 workspace 内 |

## 多人协作

每个评审者在独立分支上工作：

```bash
python run_session.py 搜索优化 --reviewer pm1 --workspace ./shared-repo
python run_session.py 风险预警 --reviewer pm2 --workspace ./shared-repo
```

分支命名规则：`review/{reviewer}/{prd_name}/{日期}`

知识库通过 Git 自动同步：评审开始前 `git pull`，结束后 `git push`。

## 故障排查

| 问题 | 解决方法 |
|------|---------|
| `ANTHROPIC_API_KEY` 报错 | 在 `.env` 中填入 key |
| `INVALID_API_KEY` 401 | 清掉环境中的 `ANTHROPIC_AUTH_TOKEN`（Claude Code 会注入旧 token） |
| `工作目录不存在` | 用 `--workspace` 指定正确路径 |
| `[blocked]` 提示 | 操作被权限围栏拦截 |
| 评审中途崩溃 | 重新运行同名 PRD，选择恢复 session |
| git push 失败 | 检查网络和 Git 凭证 |
| 飞书通知没收到 | 检查 FEISHU_WEBHOOK 是否正确 |
