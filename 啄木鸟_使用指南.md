# 啄木鸟 (Pecker) — 使用指南

> 啄木鸟专门在树上找虫。PRD 就是那棵树，bug 就是虫。
> 十只鸟各司其职：评审团并行找虫，苍鹰交叉校验，杜鹃验证质量，鸮鹦整理知识库，信鸽从下游带回反馈，伯劳审核产出。

## 前置条件

- Python 3.10+
- Anthropic API Key（支持中转站）

---

## 快速开始（3 步）

**第一步：克隆共享知识库**

```bash
git clone https://github.com/xinshu001/prd-wiki.git
```

记下克隆到的本地路径，下一步填入 `.env`。

**第二步：配置环境变量**

```bash
cp .env.example .env
```

打开 `.env` 填入以下内容：

```bash
ANTHROPIC_API_KEY=向 PM 索取
ANTHROPIC_BASE_URL=https://pikachu.claudecode.love
WIKI_PATH=填入你的本地路径
REVIEWER=你的名字
FEISHU_WEBHOOK=            # 可选
```

**第三步：安装依赖并启动**

```bash
pip install -r requirements.txt
python run_session.py PRD名称
```

评审前确保：业务资料放入 `raw/`，待评审 PRD 放入 `prd/`。

## Web 版使用

适合不使用 CLI 的同事。打开浏览器即可评审 PRD。

### 启动

```bash
cd /path/to/prd-review
streamlit run app.py
```

浏览器自动打开 http://localhost:8501

### 操作流程

1. **上传 PRD** — 拖入 .md 文件或粘贴内容，可选上传补充业务资料
2. **知识预检** — 自动扫描 wiki 知识库，显示覆盖/盲区，可填写补充说明
3. **评审** — 快速模式（Sonnet ~30s）或标准模式（4 鸟并行 + 苍鹰校验 ~2min）
4. **逐条确认** — 每条改进项可 ✅接受 / ❌驳回 / ✏️修改，附理由。支持批量操作
5. **生成报告** — 伯劳门禁检查 → 导出 3 份文档（改动报告/交互记录/差异报告）→ 一键保存到 Wiki

### 配置

- 侧边栏可修改 Wiki 知识库路径（默认 `shared-wiki/`）
- API Key 从 `.env` 文件读取，无需在页面输入
- Docker 部署：`docker compose run pecker-web`（需在 docker-compose.yml 中添加 web service）

## CLI 参数

```bash
# 自动路由（推荐）— Haiku 判断 PRD 复杂度，自动选模型
python run_session.py PRD名称

# 指定评审人
python run_session.py PRD名称 --reviewer zhangsan

# 指定工作目录
python run_session.py PRD名称 --workspace /path/to/workspace

# 强制指定模型
python run_session.py PRD名称 --model opus     # 深度评审
python run_session.py PRD名称 --model sonnet   # 标准评审
python run_session.py PRD名称 --model haiku    # 格式检查

# 关闭并行评审（单 agent 串行模式）
python run_session.py PRD名称 --no-parallel
```

## 独立工具

### 信鸽 — 反馈闭环

AI Coding 完成后运行，从代码中采集反馈信号并更新规则权重：

```bash
python feedback.py \
  --code-dir /path/to/code \
  --report output/PRD_改动报告_xxx.md \
  --prd prd/xxx.md \
  --scope keyword1,keyword2
```

`--scope` 可选，限定扫描范围（文件路径关键词过滤）。

### 杜鹃 — 评审质量验证

```bash
# 完整评测（需要测试用例文件）
python cuckoo_eval.py \
  --report output/PRD_改动报告_xxx.md \
  --test-case eval/test_cases/xxx.json \
  --workspace ./workspace

# 从已有报告生成测试用例
python cuckoo_eval.py \
  --report output/PRD_改动报告_xxx.md \
  --generate-test-case \
  -o eval/test_cases/xxx.json
```

### 鸮鹦 — Wiki 知识库维护

```bash
# 完整整理（扫描 → 修复 → Diff → 重建索引）
python kakapo_dream.py --wiki-path ./workspace/wiki

# 只扫描，不修复
python kakapo_dream.py --wiki-path ./workspace/wiki --scan-only
```

### 苍鹰 — 交叉校验

```bash
python goshawk_advisor.py --prd prd/xxx.md --report output/PRD_改动报告_xxx.md
```

### 伯劳 — 质量门禁

```bash
python shrike_review.py --workspace ./workspace
python shrike_review.py --workspace ./workspace --output review_report.md
```

检查内容：安全红线、wiki 一致性（断链/命名/frontmatter）、报告完整性。

## 环境变量说明

| 变量 | 必填 | 说明 |
|------|------|------|
| `ANTHROPIC_API_KEY` | 是 | Claude API Key，向 PM 索取 |
| `ANTHROPIC_BASE_URL` | 是 | API 地址，使用中转站填 `https://pikachu.claudecode.love` |
| `WIKI_PATH` | 是 | 共享知识库本地路径（git clone 的路径） |
| `REVIEWER` | 是 | 评审人名字，用于分支命名和报告署名 |
| `FEISHU_WEBHOOK` | 否 | 飞书群通知 Webhook，不填则跳过通知 |

## 安全机制

| 保护 | 说明 |
|------|------|
| **文件权限** | `raw/`、`prd/`、`review-rules/` 只读 |
| **bash 白名单** | 只允许 git 和 pwd 命令 |
| **危险命令拦截** | `rm -rf`、`git push -f`、`curl` 等直接拒绝 |
| **push 确认** | `git push` 需要手动确认 y/n |
| **路径围栏** | 所有文件操作限制在 workspace 内 |

## 多人协作

每个评审者在独立分支上工作：

```bash
python run_session.py 搜索优化 --reviewer pm1 --workspace ./shared-repo
python run_session.py 风险预警 --reviewer pm2 --workspace ./shared-repo
```

分支命名规则：`review/{reviewer}/{prd_name}/{日期}`

知识库通过 Git 自动同步：评审开始前 `git pull`，结束后 `git push`。

## 故障排查

| 问题 | 解决方法 |
|------|---------|
| `ANTHROPIC_API_KEY` 报错 | 在 `.env` 中填入 key |
| `INVALID_API_KEY` 401 | 清掉环境中的 `ANTHROPIC_AUTH_TOKEN`（Claude Code 会注入旧 token） |
| `工作目录不存在` | 用 `--workspace` 指定正确路径 |
| `[blocked]` 提示 | 操作被权限围栏拦截 |
| 评审中途崩溃 | 重新运行同名 PRD，选择恢复 session |
| git push 失败 | 检查网络和 Git 凭证 |
| 飞书通知没收到 | 检查 FEISHU_WEBHOOK 是否正确 |
